# System Overview

## Purpose of the System

This document provides a general overview of my home server and it's purposes, a basic description of it's configuration, and recovery and maintenance procedures.

---

## System Functions

This server is designed to be a multipurpose multi-functional environment designed to accommodate my needs and wants as an individual user and as such contains a mix of services tailored to my use, and compartmentalized through the use of containers organized into functional stacks.  At this time the server provides the following services:

- DNS routing for devices attached to the network that blocks outward communication to known advertising and metadata collection servers.
- Secure acquisitions of archival backups of physically owned or digitally licensed media through the use of public and private torrent tracker services
- A self-hosted private streaming video server, accessible both locally and externally for myself and credentialed family members to view content within my collection

---

## Major Components

- Host Operating System:
  Ubuntu Server provides the base operating system, storage management, networking, and hardware support for the server.

- Container Platform:
  Docker is used to run all major services in isolated containers, allowing services to be managed, updated, and recovered independently.

- Media Server System: 
  Jellyfin provides the media streaming server used to organize and stream video content to local and remote clients.

- Media Acquisition System: 
  Sonarr and Radarr are used to identify, select, and manage the stored location for archival TV shows and series content, and films respectively.
  qBittorrent is used as the download client for media aquisition.

- DNS Infrastructure: 
  Pi-hole provides local DNS resolution and network-wide content filtering for devices on the network.

 - VPN / Network Privacy System:
   ProtonVPN, running through the Gluetun container, provides a VPN connection used to route torrent traffic securely and prevent traffic from leaving the network outside the VPN connection.

- Backup System: 
  BorgBackup is used to back up critical system info, application data and databases for the purposes of system restore in the event of critical failures.

- Remote Access System: 
  Cloudflare Tunnel is used to provide secure remote access to internal services without exposing the server directly to the public internet.

## Data Stored on the System

The server stores several categories of data, which are separated based on their importance to system recovery and ongoing operations.

Media Library

This includes film and television content stored on the server for streaming and archival purposes. This data represents the largest portion of storage used on the system. The media library is considered replaceable, as it can be reacquired from original physical media or digital sources if necessary, and therefore is not included in critical system backups.

Application Configuration and Databases

This includes configuration files and databases for applications such as Jellyfin, Sonarr, Radarr, Prowlarr, qBittorrent, and Pi-hole. This data contains system state, metadata, library organization, indexer configuration, and operational history. This data is critical to system recovery and is included in backups.

System Configuration

This includes operating system configuration, Docker configuration, container definitions, environment configuration files, and deployment configuration stored in the infrastructure repository. This data is required to rebuild the system environment and is included in backups or stored in version control.

Download and Working Data

This includes temporary files such as active downloads, incomplete downloads, logs, caches, and temporary working files generated by applications. This data is not required for system recovery and is not included in backups.

Backup Archives

This includes Borg backup repositories containing compressed and encrypted backups of critical system configuration and application data. These backups are required for system recovery and must be preserved.

## External Services

The server relies on several external services and third-party providers for network connectivity, security, software distribution, and system functionality. These services are required for normal operation but do not store the primary system configuration or media data.

VPN Provider

A commercial VPN provider is used to provide secure network routing for specific network traffic. The VPN connection is used to route torrent traffic through an encrypted tunnel and to prevent that traffic from traversing the local ISP connection directly.

Cloudflare

Cloudflare is used to provide secure remote access to internal services through a Cloudflare Tunnel. This allows remote access without exposing the server directly to the public internet and eliminates the need for inbound port forwarding on the local network.

DNS Blocklists

Pi-hole uses externally maintained DNS blocklists to filter advertising, tracking, and known malicious domains. These lists are retrieved from external sources and updated periodically.

Torrent Indexers and Trackers

Media acquisition services rely on public and private torrent indexers and trackers to locate and download media content. These services are external to the system and are required for automated media acquisition to function.

Linux Package Repositories

The operating system and installed software rely on official Linux distribution repositories and third-party repositories to obtain system updates, security updates, and software packages.

Container Image Registries

Docker containers are downloaded from public container registries. These registries host the container images used to run system services and are required for rebuilding or updating containers.

Source Code Repository Hosting

A remote source control hosting service is used to store infrastructure configuration, documentation, and custom application code. This repository serves as the authoritative source for rebuilding the system configuration.

## Security Model

Security Model

The system is designed to minimize external exposure, isolate services from one another, protect sensitive credentials, and ensure that critical data can be recovered in the event of system failure or compromise.

Network Exposure

The server is not directly exposed to the public internet through traditional port forwarding. Remote access to internal services is provided through a secure outbound tunnel, which allows external access without opening inbound ports on the local network. Only services intended for remote access are made available through this tunnel.

Service Isolation

All major services run inside Docker containers. This provides isolation between services so that a failure or compromise in one service does not automatically compromise the entire system. Containers can be updated, rebuilt, or removed independently of the host operating system.

VPN Traffic Isolation

Torrent traffic is routed through a dedicated VPN container that functions as a network gateway for the torrent client. The VPN container includes a network kill switch that prevents traffic from leaving the system if the VPN connection is interrupted. This ensures that torrent traffic does not leave the network outside of the encrypted tunnel.

Credential and Secret Management

Sensitive credentials such as VPN credentials, API keys, and backup encryption keys are not stored in source control. These secrets are stored in environment files on the server with restricted file permissions and are documented separately so they can be recreated during system recovery.

Access Control

Administrative access to the server is restricted to local network access and secure shell (SSH). Application-level access to services such as Jellyfin and Pi-hole requires authentication. Remote access to services is controlled through the tunnel provider’s authentication and access controls.

Backup Security

Backups are encrypted and stored in a format that prevents unauthorized access to backed-up data. Backup repositories are protected by encryption keys that are stored separately from the backup data itself.

Principle of Least Exposure

Only services that must be accessible are exposed to the network. Internal services communicate over the internal Docker network and are not exposed externally unless required for functionality.

## Backup and Recovery Summary

Backup and Recovery Summary

The system is designed so that it can be rebuilt from a combination of version-controlled configuration files and encrypted backups of critical system data. The backup and recovery strategy focuses on preserving system configuration and application state rather than large media files, which can be reacquired if necessary.

Backup Scope

Backups are performed for critical system data required to restore the server to operational status. This includes:

Application configuration files and databases
Container configuration and environment files
System configuration files
Custom scripts and automation tools
Documentation and infrastructure configuration stored in the repository

Large media files and temporary download data are not included in backups because they can be reacquired or regenerated.

Backup Method

Backups are performed using an encrypted backup system that creates compressed, versioned snapshots of critical system data. These backups are designed to allow restoration of individual files or full system configuration as needed.

Recovery Strategy

System recovery is designed around the concept of rebuilding the server from documented configuration and restoring application state from backups. The general recovery process is:

Reinstall the operating system.
Restore the infrastructure repository and configuration files.
Recreate required environment and secret files.
Restore application configuration and databases from backup.
Rebuild and start containers.
Verify system functionality.

This approach ensures that the system can be restored to a known working state even after complete hardware failure.

Documentation

Detailed backup procedures are documented in the Backup Strategy document.
Detailed system rebuild procedures are documented in the Recovery Plan document.

This backup and recovery model ensures that critical system configuration and state are preserved while avoiding unnecessary backup of large, replaceable media files.
